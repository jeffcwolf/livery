Design Issues — release-scholar
3 functions exceed the 50-line threshold. scan_git_history at 70 lines handles commit walking, tree diffing, pattern matching, and result accumulation — four distinct concerns in one function.

Mutable shared state as interface. Every function takes &mut Report. Side effects are the contract. Functions cannot be called without a live Report object and cannot be tested in isolation.

Concern leakage. audit_gitignore() reports to the Gitignore category from inside security.rs. Two categories share one file — a boundary violation.

Zero doc comments on all 8 functions. No false-positive risk notation on regex patterns.
Design Strengths — mint
All 9 functions under 36 lines. Each does one thing. The longest, check_secrets_in_history, handles exactly one check and returns exactly one value.

Value-returning interface. Every check function returns a CheckItem. No shared state. Functions are independently composable and testable without any framework.

Strict concern separation. Security is security. Gitignore has its own category and its own file. No leakage between them.

7 doc comments including inline false-positive risk annotations on regex patterns — e.g. "Risk: moderate false-positive rate on test fixtures. Mitigated by requiring keyword proximity."

/// Scan git history for secret patterns.① doc comment states contract
/// Skipped if `config.no_history()` is true.
fn check_secrets_in_history(ctx: &ProjectContext) -> CheckItem {② outcome verb · returns value · no mutation
    if ctx.config.no_history() {③ predicate accessor · reads as English
        return CheckItem {
            name: "no-secrets-in-history".into(),
            status: CheckStatus::Pass,
            message: "history scan skipped (no-history flag set)".into(),④ skip reason propagated to caller
        };
    }

    let output = Command::new("git")⑤ subprocess · no C library dependency
        .args(["log", "-p", "--all", "-100"])
        .current_dir(&ctx.project_dir)
        .output();

    let Ok(output) = output else {
        return CheckItem { name: "no-secrets-in-history".into(),
            status: CheckStatus::Fail,
            message: "could not run git log".into() };
    };

    let diff_text = String::from_utf8_lossy(&output.stdout);
    let secret_patterns = build_secret_patterns();⑥ named constructor · independently testable

    for (pattern, description) in &secret_patterns {
        if pattern.is_match(&diff_text) {
            return CheckItem { ..., message: format!("...{description}") };⑦ description propagated · caller gets full context
        }
    }
    CheckItem { name: "no-secrets-in-history".into(),
        status: CheckStatus::Pass,
        message: "no secret patterns found in git history".into() }
}
ARC · Ousterhout Compliance
① Doc comment states purpose and the one conditional behaviour — no ambiguity for the reader.
② Name check_secrets_in_history: verb + object + context. Return type CheckItem makes the outcome explicit in the signature.
③ no_history() is a predicate accessor — the if reads as natural English.
④ Skip reason is carried in the return value — caller can display it; nothing is lost.
⑤ Git subprocess replaces git2 (libgit2 C binding) — no native dependency, same result, simpler build.
⑥ build_secret_patterns() is a named constructor — patterns can be tested independently from the scan logic.
⑦ Pattern description propagates to the caller — "private key header found" rather than "something was found."

Industry baseline for well-tested code is typically 20–30% survival. Mint's security module at 17% is above average. The 3 missed mutants are specific, named, and fixable — see below.
3 Missed Mutants
Each reveals a real gap in the test suite. Each has a one-test fix.
list_tracked_files security.rs:48 · match guard
replace match guard output.status.success() with true
The git subprocess failure path is untested. If git returns a non-zero exit code, the mutation replaces the success check with true — meaning a failed git invocation would now be treated as success and return whatever garbage is in stdout. No test catches this because no test exercises the failure branch of output.status.success().
Fix: Add a test that calls list_tracked_files on a directory that exists but is not a git repository (after the exists() guard passes). Git will exit non-zero. Assert the function returns Err(ArchiveError::NotAGitRepository) or equivalent, not an empty vec.
gitignore_contains security.rs:152 · boolean operator
replace || with && in gitignore_contains
The predicate trimmed.is_empty() || trimmed.starts_with('#') filters out blank lines and comment lines. Flipping || to && means only lines that are both empty and start with # are skipped — which is impossible, so neither blank lines nor comment lines are ever skipped. The existing tests don't use gitignore content that contains both a comment line and an empty line independently.
Fix: Add a test where the gitignore contains # a comment on one line and an empty line on another, with the target pattern on a third line. Assert the function returns true — confirming the comment and empty lines were skipped correctly and the pattern was found.
check_secrets_in_history security.rs:182 · negation
delete ! in check_secrets_in_history
The guard if !output.status.success() handles the case where git log fails (no history, detached HEAD, etc.) by returning Pass with "no git history to scan." Removing the ! inverts this: a successful git log run is now treated as "no history" and returns Pass immediately, skipping all pattern matching. The existing no_history_flag_skips_history_scan test only exercises the early-return path, not the success/failure distinction.
Fix: Add a test where git log succeeds (a real commit exists, no_history is false) and the content is clean. Assert the function returns Pass with "no secret patterns found" — not "no git history to scan." This forces the success branch to be exercised distinctly from the failure branch.

15 of 18 meaningful mutants caught. The 3 missed mutants are not random noise — they cluster on failure-path branches (git subprocess failure, boolean logic in a predicate, inverted success guard). This is a recognisable pattern: TDD naturally produces good coverage of the happy path and the explicitly tested failure modes, and undercovers the implicit failure branches that weren't in the original test specification.

All 3 gaps are one test each to close. The property test on build_secret_patterns caught mutations that unit tests alone would have missed — demonstrating that the two test types are complementary, not redundant.
