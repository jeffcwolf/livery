Two lessons from Phase 0:

Lesson 1 — Submodule initialization is not automatic. Claude Code clones the repo but does not run git submodule update --init --recursive. If Livery is a submodule, the livery/ directory will be empty in the container. The project CLAUDE.md must include an explicit instruction to check for and initialize the submodule before reading any Livery files. This should be the very first step in the pre-flight block, and it should be added to the project.md.template in Livery so every new project gets it by default.

Lesson 2 — "Read this file first" is not a reliable instruction. Claude Code auto-loads CLAUDE.md at session start, but a passive note like "Extends: read CLAUDE-base.md first" inside that file is advisory, not guaranteed. The agent may skim it and proceed without loading the referenced file. The pre-flight block needs to be formatted as a mandatory numbered procedure — visually prominent, action-oriented, impossible to skip — not a quiet metadata line. The pattern that works: "MANDATORY PRE-FLIGHT" header, numbered steps, explicit file paths, concrete actions.

**Lesson 3 — Claude Code timeouts silently degrade output quality.** When Claude Code hits a timeout during a multi-step session, it does not resume where it left off — it either skips the remaining steps or produces abbreviated versions of them. The developer has no clear signal that steps were skipped. In our Phase 1 session, timeouts caused the agent to omit concrete Rust function signatures (replacing them with prose descriptions), skip the standards audit entirely, drop the required Mermaid dependency graph, and produce ADRs in informal prose rather than the required structured format. The resulting ARCHITECTURE.md looked complete at a glance but was missing the precise contracts that make the document useful during implementation. A follow-up session was required to fill the gaps. The implication for Livery: every phase prompt should end with an explicit gate checklist that the agent must self-verify before declaring the phase complete, and the developer should verify that checklist against the output before proceeding to the next phase. Timeouts are not visible failures — they produce plausible but incomplete work, which is harder to catch than an outright error.
